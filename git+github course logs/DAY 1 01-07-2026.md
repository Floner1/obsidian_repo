Git is version control software, keeps track of multiple versions of any file, code, videos, images, etc. 

github is where u merge different versions of the same program into one place

git is divided into 2 parts, local, remote. local is on your device, remote is in cloud, where files are uploaded. 
remote is usually referred to as github

repository: place where all version of files and complete change history are stored. 

steps:
1. u work in your local repository
2. u stage your changes (telling git that u wanna keep these changes) and review, adjust,remove changes them before committing
3. commit changes, permanently saves them to local repo, only on pc, not on cloud yet

TERMINAL (POWERSHELL/CMD) COMMANDS:
cd = change directory, move to new folder.
cd folder name to move forward, cd.. to move backwards. 

pwd = print working directory
will print the absolute path of the current folder u are working in

mkdir = make directory
will make new folder/directory
mkdir (options) name
mkdir name will just make new folder inside ur current directory
mkdir -p name/name/etc will make nested folders

GIT COMMANDS (windows specific):
git init, starts tracking folder
if git is tracking ur folder, u type ls to display all files and folders inside the folder that git is tracking


Get-ChildItem -Force lets see hidden file in system

git clone (repository url), copy url from github down into the folder that u want

git status: allows us to see what file has been changed/updated

git add: moving files from working directory to staging is called adding basically telling git that i wanna keep this change that ive made to the file
u can use git add -A/-all to add everything in the current folder that is being tracked to stage
![[git tracking flow.png]]

scope of reset is commits, not specific files
git reset: unstage everything that you have just staged, but will only display that previously tracked files have been reseted. untracked files that got stagged but have never been commit will not be displayed as unstaged, just silently unstaged.

git add . : only stage files in ur current working directory and its subdirectories, not files in all directories

git add * : stage all new or modified files, no deleted ones

git add filename or git add directory: only stage specified file/directory

git add * . extension type: stages all files with that specific extension, excluding deleted ones

git commit -m "message": commits to github and also includes a message
if it is first time committing then git might ask to confirm identity
if ask to confirm identity, do this:
git config --global user.email "email address"
git config --global user.name "any name u want"
these 2 commands sets email and name globally, every git commit made from this device will use specified email and name
GIT COMMIT ONLY COMMITS TO LOCAL REPO, NOT GITHUB

if u just wanna set for local directory, then use --local instead of --global.

git reset HEAD~ : undo last commit and bring everything back to working repo
head is the pointer towards commit you are currently on

git rm filename - delete file and stage that deletion simultaneously
rm is remove

git reset --hard: unstage and restore all files that u have deleted. 

if u delete smth, then use reset --hard and then edit the thing u just restored, this msg will be thrown ![[reset --hard error msg.png]]
this is cus the staging area and the working directory dont match, reset --hard makes those 2 match exactly to each other, detected uncommitted changes in the file

to remove, u gotta commit changes or confirm that u wanna delete.
if u rlly wanna delete, then git rm -f (or --force) filename

GIT RM WILL ONLY TOUCH TRACKED FILES

git rm --cached filename: removes file from staging area, but keeps file in directory, staging area shows file as pending deletion. u gotta commit it for it to truely stop tracking
File sits on disk as an ordinary untracked file. Add a .gitignore entry if you don't want git nagging you about it showing up in status.
![[--force vs --cached.png]]

git rm -r folder: remove folder and every tracked file

git log (--oneline if u want short summary): view commit history. u can use commit ids to trace back to prev commits

BRANCHING:
create seperate dev branch from main branch to test/review/manage changes and commit them in that branch to see if work or not. once stable, u can merge that branch back to main. 

git branch: see branches

git branch branchname: create new branch
star on branch means we are on that branch

whenever u create a new branch, it takes the current state of the branch u are currently on and duplicate.
![[branching flow.png]]
git checkout branchname: moves into specified branch. when u type git status and u have created branches, it will say what branch u on

if u made smth in dev branch, and switch back to main, then it will hide what is in dev branch. 

git merge branchname -m "message": merges the specified branch with ur current one.

git checkout (commit id): brings u back to any commit id u want, latest commit is still safe and accessible

If you want to create a new branch from the commit that u just switched to to retain commits you create, do this:
  git switch -c <new-branch-name>
Or undo this operation with:
  git switch -

if u wanna quick go back to most recent, just go git checkout branchname

Compare commits:
git diff commitid1 commitid2: compares what changed between the 2 commits
most recent one first, older one second, perspective of newer commit
u can do reverse the order and do vice versa
removed in red, added in green

GIT PUSH: send local changes to remote
git fetch: bring remote changes to local, but not merge
git pull: fetch + merge, working directory will immediately reflect remote changes, used when remote has new commits u dont have. 

git push origin main: origin is remote repo, main is the branch u are pushing to. 
u push the branch that u are currently on
so if u push staging, it will create new branch on remote

GIT RESTORE:
restore to previous versions
git restore filename: restore to specified file to latest committed version
git restore commitid: restore entire code to latest commit
git restore . : restore every file to latest commit
git restore  --staged . to unstage everything

when to use restore vs reset:
Want to undo edits in a specific file without messing with commit history → restore.
Want to move your entire branch back to an earlier commit, potentially rewriting what's staged or on disk → reset.

GIT STASH
git stash: temporarily set aside unfinished work to switch to diff branch
after u finished and u go back to ur original branch, u do git stash pop to restore the temporarily saved changes
if u stashed multiple things, git stash pop will bring back most recently stashed thing
git stash apply is same as git stash pop, but then doesnt remove the restored code from the stash 
git stash list to see list of things that are currently stashed
u can apply or pop specific stashes, just specify with the stash id

GIT REVERT:
undo changes of last commit, but doesnt delete last commit, just creates a new commit with the code of old commit

reset is diff bc reset brings back to specific commit and discard all commits after that, but revert doesnt discard, just brings back

GIT REBASE:
basically merge, but then it doesnt add missing code from main onto branch, but then it moves branch so that it matches the newest commit of main. prevents commit log from getting messy bc rebase doesnt produce a commit.
